# 要解决的问题/痛点
1. 问题一：用户在使用Agent产品/cli的时候，常常忽略自己的提示词中前后矛盾的地方，导致上下文污染
2. 问题二: skill太多，plugin太多，使用时会相互冲突，上下文污染，还导致加载过慢的问题，用户也很难肉眼修复

# 适配架构
1. opencode
2. codex
3. claude code
4. cursor
5. openclaw

# 使用方式
用户直接用指令驱动
/contextdoctor 检查当前上下文的污染，输出html可视化报告
/repair 不仅检查当前上下文的污染，输出html可视化报告，还提供解决方案

# 特性
1. 极简安装，快速上手
2. 易于改装，框架灵活

# README要求
Readme多语言支持，打造GitHub仓库品牌，增强宣传推广效果。
Readme包含报告模板的截图。

# 污染类型
1. skill, plugin污染，互相冲突，或冗余
2. 需求冲突，前后用户指令不一致
3. 某个位置的错误，可能导致错误的累积

# 可视化报告                                                                                                   
                                                                                                              
## 基础要求                                                                                                    
1. 支持生成多种主流语言的报告（zh/en/ja/ko）。                                                                 
2. 报告内嵌语言切换器，支持客户端无刷新切换，语言偏好持久化到 localStorage。                                   
3. 严重等级三级划分：Critical / Warning / Suggestion，用色彩系统区分（见 DESIGN.md）。                         
4. 报告保存在固定路径 ~/.contextdoctor/reports/，文件名格式：                                                  
  context-doctor-{type}-{timestamp}-{lang}.html                                                               
                                                                                                              
## 内容要求                                                                                                    
5. Header 区域必须包含以下元数据：                                                                             
  - 报告生成时间                                                                                              
  - 检测维度列表（skill冲突 / 需求冲突 / 错误累积）                                                           
  - 检测到的对话轮次数量（若可获取）                                                                          
6. 评分必须有明确的扣分依据展示（每条 issue 对应的扣分权重），                                                 
  不得出现无来源的分数。                                                                                      
7. 所有统计数字（Critical数、Warning数等）必须与实际 issue 列表严格对应，                                      
  禁止使用写死的占位数据。                                                                                    
                                                                                                              
## 条件渲染规则（关键）                                                                                        
8. 当 issue 总数为 0 时：                                                                                      
  - 隐藏 Distribution 图表区域（空柱状图无信息量）                                                            
  - 展示"检测通过"状态，并列出所有已检测通过的维度，给用户正向反馈                                            
  - Recommendations 区域内容不得是泛泛套话，应具体说明"当前上下文的哪些特征                                   
    使其保持健康"                                                                                             
9. 当 issue 总数 > 0 时，Distribution 图表必须有实际高度，                                                     
  柱子高度为 0 的维度不渲染对应柱子。                                                                         
                                                                                                              
## 视觉规范                                                                                                    
10. 禁止使用 emoji 作为功能性 UI 元素（状态指示点、图表、图标等）；                                            
   emoji 仅允许出现在纯装饰性场景（如 empty state 的插画位置）。                                              
   状态指示器必须用 CSS 实现（colored dot、badge 等）。                                                       
11. Health Trend 指标仅在存在历史报告文件可供对比时才显示，                                                    
   否则隐藏该指标，不得显示无意义的 `→` 占位符。                                                              
12. 满分（≥90）状态下，score circle 的颜色必须与 header 背景色有足够对比度，                                   
   确保视觉可识别。

# 具体实现方式
1. 用户运行对应指令的时候，将指令直接转化为对于当前Agent的指令（让Agent反思自己当前指令中是否有上下文污染的情况，给出综合评分，按照给定的报告格式，输出可视化报告）
2. 安装的时候，直接将指令全局安装，用户在任何位置使用Agent框架，都可以使用上下文检查指令。
3. 指令的实现方式需要参考每个具体的 Agent框架 里面定制指令的方式（见COMMANDS_REFERENCE.md）。
4. 所有框架采用当前窗口默认model作为doctor，不特殊指定
5. 注意文档的安装方式采用本地安装方式
6. 实际报告生成方式需要明确采用同一套html模板，都使用同一套路径，可以用不同框架来区分，命名方式也保持一致。
7. 默认行为，每次Agent完成可视化报告生成后，使用open命令直接打开
8. 需要在Agent提示词中说明，以及设计这个插件的时候需要注意，每次Agent生成可视化报告的时候，禁止直接用脚本的方式进行诊断，而是必须自行根据上下文进行分析，之后利用脚本来输出相应格式的报告
